Clean-room 表示复核者不继承作者的需求基线、代码结论、自查结果和判断锚点，但仍直接读取全部有效原始需求。

## 1. 前置条件

启动复核前必须具备：

- 全部有效原始需求及其状态、顺序和用户明确关系；
- 可直接读取的原始图片和视觉附件；
- 可访问的工作区或用户指定代码范围；
- 可获得的代码快照信息；
- 完整待审需求文档。

关键一手来源不能让新审查者访问时，不得用作者摘要替代，也不得声称完成独立复核。

## 2. 上下文隔离

启动一个全新、未参与主分析或上一轮审查、具备只读搜索和视觉读取能力的审查实例。判断标准是实际上下文隔离能力，不绑定具体产品或工具名称。

运行时适配说明：在 Claude Code 等支持子代理的环境中，可用限定为只读工具集的全新子代理充当审查实例；其他环境以能否满足上述隔离标准为准。

可以提供：

- 有效原始来源定位符或逐字正文；
- 来源状态及用户明确建立的覆盖、撤回和补充关系；
- 工作区、用户指定代码范围和代码快照；
- 完整待审文档；
- 本协议固定任务和报告格式。

保持以下作者材料在隔离边界之外：

- 作者整理的需求基线和追溯矩阵；
- 作者认定的项目、代码边界、调用关系和实现结论；
- 作者搜索记录、自查结论和预期 verdict；
- 之前审查者的问题清单和结论；
- 与需求无关的历史对话。

用户建立、修改、撤回或解释需求范围的原话属于原始来源，必须保留。

## 3. 固定审查任务

使用以下自包含、中性任务：

```text
你是一名首次接触作者分析过程的 clean-room 独立需求审计员。本次审查只读，不修改、创建、删除或格式化任何文件，也不运行会改变工作区的操作。

事实材料：
1. 有效原始需求：<来源清单、定位符或逐字正文及明确关系>
2. 本地代码：<工作区和用户指定范围>
3. 代码快照：<branch/commit/version/时点>
4. 待审需求文档：<完整正文或稳定定位符>

请从头独立完成：
A. 从全部有效来源提取目标、保持不变项、名称、文案、对象、属性、数值、单位、条件、顺序、时间、交互、边界和验收要求。未裁决冲突不得自行决定。
B. 独立识别需求指向的项目或组件，核验它与本地代码是否一致。
C. 重新搜索实际生效的实现来源，不信任待审文档中的路径、代码片段和调用关系。
D. 逐项对照原始需求、实际代码和待审文档，检查遗漏、改变、无依据扩展、错误代码事实、现状/目标混淆、追溯和验收不一致及不可验证声明。
E. 只输出审查报告，不提出需求外优化，不直接修改文档。

每个问题必须包含文档位置、原始需求证据、代码证据、问题说明和不扩大需求的最小修正方式。证据不足时使用 UNVERIFIED 或 BOUNDARY 规则。
```

## 4. 问题分类

- **BLOCKER**：项目不一致、关键实现缺失却声称完整、来源冲突未解决、文档实质改变需求或当前不可安全交付。
- **MAJOR**：遗漏明确需求、加入无依据要求、关键代码事实错误，或规则、文案、数值、条件和验收明显不准确。
- **MINOR**：不改变业务含义，但证据定位、表达或追溯不够准确。
- **UNVERIFIED**：规范性声明或事实结论无法由可用材料验证；任何 `UNVERIFIED` 都阻断 PASS。
- **BOUNDARY**：缺失内容与核心需求无关，文档没有作确定声明且已披露；它不计为问题。

## 6. Verdict 规则

### PASS

只有全部满足才输出：

1. 所有有效原始来源均可访问并已审阅。
2. 不存在未解决来源冲突。
3. 项目一致性成立。
4. 代码完整或部分完整但足以成文。
5. 所有目标、保持不变项和验收标准都有原始需求依据。
6. 所有代码现状结论都有权威实现证据。
7. `BLOCKER=0`、`MAJOR=0`、`UNVERIFIED=0`，且没有影响准确性的未解决 `MINOR`。
8. 需求证据、代码快照和待审文档版本绑定有效。

`BOUNDARY` 可以存在，但不得影响核心需求，文档也不能对缺口作确定声明。

### FAIL

现有材料足够，但文档存在可以根据证据修正的问题。

### NEEDS_INPUT

必须由用户或外部来源补充、恢复或裁决材料，例如关键来源不可访问、图片无法辨认、来源冲突、项目身份无法确认、关键实现缺失或需求含义存在多个会改变结果的合理解释。

## 7. 问题闭环与失效

作者逐条核验审查证据：

| 问题编号 | 处理 | 依据 |
|---|---|---|

- 证据成立：执行满足原始需求所需的最小修正。
- 证据不成立：保留文档并记录具体反证。
- `NEEDS_INPUT`：停止最终交付并提出聚焦问题。
- `FAIL`：文档保持草稿状态。

任何被复核载荷修订后，使用另一个全新审查者重新审查完整文档。需求来源、来源关系、代码快照或被复核载荷内容变化，或发现之前不可访问的一手材料，都会使旧 `PASS` 失效。复核完成后按事实写入或更新复核证明元数据（复核状态字段和“独立复核结果”章节）不属于载荷修订，不使 `PASS` 失效。

完整复核累计不超过 3 轮。超过上限仍未 `PASS` 时，不再自动进入新一轮，转为 `NEEDS_INPUT`，向用户报告未收敛的问题清单并请求裁决。

工具不能保证隔离或完整传递一手材料时，明确说明复核未完成，需求文档只能标记为草稿。
