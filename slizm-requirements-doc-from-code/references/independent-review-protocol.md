# Clean-room 独立复核协议

本协议用于对需求文档进行与主代理分析隔离的独立审查。clean-room 不代表看不到原始需求，而是复核者不继承作者的需求基线、代码结论、自查结果和判断锚点。

## 1. 前置条件

启动复核前，必须具备：

- 全部有效原始需求来源及其状态、顺序和用户明确关系
- 可直接读取的原始图片或视觉附件
- 可访问的本地工作区或用户指定代码范围
- 可获得的代码快照信息
- 完整待审需求文档，使用文件定位符或完整正文

若关键一手来源无法让新代理访问，不得用主代理摘要替代，也不得声称完成独立复核。

## 2. 上下文隔离

启动一个确实创建隔离上下文的全新审计代理。在 ZCode 中使用新的、具备读取与搜索能力的通用代理调用；不要复用参与过主分析或上一轮审查的代理，也不要向已有代理补发上下文。

若当前运行环境不能保证新代理与主分析隔离，明确说明限制，待审文档只能标记为草稿。

### 可以提供

- 有效原始来源的定位符或逐字正文
- 来源状态和用户明确建立的覆盖、撤回、补充关系
- 当前工作区和用户指定代码范围
- 分支、commit、版本或等价代码快照信息
- 完整待审文档路径或完整正文
- 本协议中的固定审查任务和报告格式

### 不得提供

- 主代理整理出的需求基线或追溯矩阵
- 主代理认定的项目、代码边界、调用关系和实现结论
- 主代理搜索过哪些文件、认为哪些证据最重要
- 主代理自查结论、问题处理倾向或预期 verdict
- 之前审查代理的问题清单和结论
- 与需求无关的用户偏好或闲聊

用户建立、修改、撤回或解释需求范围的原话必须保留，不能因为它来自“之前对话”而排除。

## 3. 固定审查任务

使用以下自包含、中性任务。根据当前环境替换定位符或直接嵌入完整正文，不添加希望它通过或证明某结论的引导。

```text
你是一名首次接触作者分析过程的 clean-room 独立需求审计员。不要默认待审文档正确，也不要推测作者意图。

本次审查只读。不要修改、创建、删除或格式化任何文件，也不要运行会改变工作区的操作。

你只能依据以下一手材料和固定审查规则：
1. 有效原始需求来源：<来源清单、可访问定位符或逐字正文>
   - 逐项读取所有 active 来源及 partially-superseded 来源中的有效片段。
   - 按用户明确记录的补充、替换、撤回和部分覆盖关系处理。
   - 不得假设较晚来源自动覆盖较早来源。
   - 图片必须使用当前环境的视觉读取能力直接查看原图；OCR、转录和作者图片描述只能辅助。
2. 本地代码：<工作区及用户指定范围>
3. 代码快照：<branch/commit/version/时点>
4. 待审需求文档：<文件定位符或完整正文>

请从头独立完成：
A. 从所有有效来源提取明确目标、保持不变项、名称、文案、对象、属性、数值、单位、条件、顺序、时间、交互、边界和验收要求，并引用 SRC-nnn。发现未裁决的来源冲突时不要自行决定。
B. 独立识别需求指向的项目或组件，核验它与本地代码是否一致。不要把用户指定范围或待审文档结论当作身份成立的证据。
C. 从头搜索并读取足以验证需求相关现状的实际生效实现来源，不信任待审文档中的路径、代码片段和调用关系。
D. 逐项对照原始需求、实际代码和待审文档：
   - 是否遗漏、弱化、强化或改变原始需求
   - 是否加入无原始依据的产品或技术要求、保持不变项或验收项
   - 当前现状、标识符、规则、执行关系、状态/数据/配置/资源来源是否真实
   - 是否混淆需求目标与代码现状
   - 项目一致性和代码充分性结论是否成立
   - 追溯矩阵和验收标准是否准确且没有扩大范围
   - 是否把不可验证内容写成确定事实
E. 不要提出超出原始需求的优化，不要直接修改文档，只输出审查报告。

每个问题必须包含：
- 待审文档位置或原句
- 原始需求证据：SRC-nnn + 原句/文档位置；图片使用文件名和可定位区域
- 代码证据：文件/资源定位 + 行号、符号、配置项或可定位片段
- 问题说明
- 不扩大需求的最小修正方式

若没有足够证据证明问题存在，不要断言；按下面的 UNVERIFIED 或 BOUNDARY 规则处理。
```

## 4. 问题分类

- **BLOCKER**：项目不一致、关键实现缺失却声称完整、需求来源冲突未解决、文档实质改变需求，或当前不可安全交付。
- **MAJOR**：遗漏明确需求、加入无依据要求、关键代码事实错误、规则/文案/数值/条件/验收明显不准确。
- **MINOR**：不改变业务含义，但证据定位、表达或追溯不够准确。
- **UNVERIFIED**：文档提出了无法由可用原始材料或代码验证的规范性声明或事实结论。任何 `UNVERIFIED` 都阻断 PASS。
- **BOUNDARY**：缺失内容与核心需求无关，文档没有对其作确定声明，并已如实披露证据边界。它不是问题，也不计入 `UNVERIFIED`。

## 5. 固定报告格式

```markdown

- MAJOR: <数量>
- MINOR: <数量>

- BOUNDARY: <数量>
- 一句话结论：

- 一致性结论及证据：
- 代码充分性结论及缺口：
```

## 6. Verdict 规则

### PASS

只有全部满足才输出：

1. 所有有效原始来源都可访问并已审阅。
2. 不存在未解决的来源冲突。
3. 项目一致性成立。
4. 代码达到“完整”或“部分完整但足以成文”。
5. 所有目标要求、保持不变项和验收标准都有有效原始需求依据。
6. 所有代码现状结论都有实际权威实现证据。
7. `BLOCKER=0`、`MAJOR=0`、`UNVERIFIED=0`。
8. 不存在影响准确性的未解决 `MINOR`。
9. 需求证据版本、代码快照和待审文档版本与报告绑定一致。

`BOUNDARY` 可以存在，但必须不影响核心需求，且待审文档没有对缺口作确定声明。

### FAIL

现有材料已经足够，但待审文档存在可以根据证据修正的问题。

### NEEDS_INPUT

必须由用户或外部来源补充、恢复或裁决材料才能继续，例如：

- 关键原始来源不可访问
- 图片无法可靠辨认
- 有效来源之间冲突且无明确覆盖关系
- 项目身份无法确认
- 关键实现缺失
- 需求含义存在多个会改变结果的合理解释

## 7. 主代理处理审查结果

子代理结论不是权威本身，证据才是。逐条复核问题证据：

| 问题 | 处理 | 依据 |
|---|---|---|
| 编号 | 已修正 / 不成立 / 待用户确认 | 原始来源或代码证据 |

- 证据成立：只做满足原始需求所需的最小修正。
- 证据不成立：不修改文档，并记录具体反证。
- `NEEDS_INPUT`：停止交付并向用户提出聚焦问题。
- `FAIL`：不得称为最终版。

任何文档修订后，启动另一个全新的 clean-room 代理，使用同一有效原始来源和当前代码快照重新审查完整文档。不要把上一轮报告交给新代理，不复用原代理，也不只审查差异。

## 8. PASS 失效条件

以下任一情况发生后，旧 `PASS` 自动失效：

- 需求来源内容、状态、关系或有效片段变化
- 本地代码、配置、生成来源、分支或版本变化
- 待审需求文档发生任何修改
- 发现之前不可访问的一手材料

失效后按当前版本重新执行完整 clean-room 复核。

若工具不可用、上下文不能隔离或一手材料无法传给新代理，明确说明复核未完成，需求文档只能标记为草稿。
