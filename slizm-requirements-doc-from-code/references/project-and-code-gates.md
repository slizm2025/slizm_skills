# 项目一致性与代码充分性门禁

本协议用于确认需求是否指向当前本地代码，以及本地实现是否足以支撑准确成文。按实际项目形态动态分析，不套用固定架构。

## 1. 识别候选代码单元

先观察工作区实际组织方式。目标可能是单体、多模块、多仓库、公共库、命令行工具、桌面或移动应用、嵌入式程序、插件、数据流程、模型、声明式配置、基础设施定义或其他形态。

可根据实际存在的内容检查：

- 项目、产品、组件、仓库、模块、包或命名空间名称
- 项目说明、构建与依赖清单、入口、公开契约、配置和资源
- 需求中的专有术语、标识符、路径、命令、协议、数据结构或可观察行为
- 输入输出、状态变化、持久化、生成过程或外部依赖
- 测试、运行、部署和发布描述
- 与其他代码单元、系统或环境的关系

这些是可选维度，不是固定检查表。工作区包含多个独立代码单元时分别列出候选项。

## 2. 项目一致性门禁

分别记录：

- **用户指定范围**：用户希望分析的位置。
- **代码身份核验**：本地证据表明该位置实际是什么。
- **一致性结论**：需求与本地代码是否对应。

用户指定范围不能覆盖明显的身份冲突。

### 一致

满足以下任一条件，且不存在反向证据：

- 有一个决定性证据能够唯一对应需求和代码；或
- 有多个相互独立的强佐证共同建立对应关系。

决定性或强证据可能包括：

- 需求中的全局唯一标识、公开契约或资源在实现中确实存在
- 需求指向的入口与相关行为可以形成实际生效路径
- 项目元数据明确声明相同用途、产品或组件身份
- 需求中的多个专有对象和规则在同一候选代码单元中吻合

目录名相似、单个通用词语相同或待审需求文档声称某路径存在，都不是充分证据。

### 不一致

出现明确反向证据，例如项目用途、业务域、公开契约、核心标识或可观察行为与需求冲突。

### 无法确认

存在多个同样合理的候选项，或证据不足以唯一定位。

### 阻断提问模板

```text
当前无法确认需求与本地代码属于同一目标：
- 用户指定范围：<未指定或用户提供的位置>
- 需求指向：<项目、组件、用途或关键行为线索>
- 本地识别：<候选代码单元及其实际用途>
- 不匹配或缺失证据：<关键事实>

请明确本需求对应的代码位置；若实现分布在多个代码单元，请按实际职责提供相关目录、仓库、模块或组件。
```

结论不是“一致”时，停止代码深挖和需求文档成文；已经完成的原始需求来源登记可以保留。

## 3. 从需求推导必需实现事实

项目一致后，不按技术分层机械检查。先逐条询问：

- 为确认当前行为，需要找到什么触发或使用方式？
- 哪些实际实现或声明负责关键判断、转换、计算、编排或状态变化？
- 相关数据、状态、配置、资源或依赖从哪里取得？
- 结果如何输出、保存、传递、生成或产生副作用？
- 哪些测试、契约、配置、部署定义或生成规则可以证明当前行为？

只检查与原始需求事实判断直接相关的内容。

## 4. 权威实现来源

优先读取实际生效的权威实现来源，而不局限于“可执行代码”：

- 运行时代码和调用关系
- 声明式配置和基础设施定义
- schema、契约、规则文件和生成来源
- 构建、部署或运行环境实际消费的配置
- 数据转换、模型或工作流定义
- 证明当前行为的测试和固定样例

README、注释、历史设计和示例属于辅助证据。它们与实际生效来源冲突时，明确记录冲突，不静默采用任一方，也不由此扩展需求。

## 5. 代码充分性门禁

### 完整

能够确认需求涉及的所有现状结论、核心规则、事实来源、执行关系和必要边界。

### 部分完整但足以成文

只有同时满足以下条件才能使用：

1. 写入需求文档的每个代码现状结论都有实际证据。
2. 缺失内容不会改变任何明确需求的业务含义。
3. 缺失内容不会改变规范性验收标准。
4. 文档不会把未分析部分伪装成完整影响范围。
5. 缺口已经作为证据边界披露，而不是由假设填充。

### 关键实现缺失

缺失会导致以下任一内容只能猜测：

- 当前规则或可观察行为
- 数据、状态、配置或资源来源
- 需求与现状的关系
- 需求相关代码边界
- 验收含义

### 阻断提问模板

```text
已确认当前工作区包含目标代码，但尚缺少支撑准确需求分析的关键实现：
- 已能确认：<事实及证据位置>
- 尚不能确认：<缺失的行为、规则、来源、依赖或边界>
- 缺失影响：<因此无法可靠判断的需求内容>

请提供负责上述事实的代码位置；若它位于其他仓库、模块、组件、生成来源或运行配置中，请告知对应位置和职责。
```

## 6. 分析与引用

按真实项目形态建立证据路径：

```text
需求项
→ 触发或使用方式
→ 关键实现或声明
→ 相关状态、数据、配置、资源或依赖
→ 可观察结果或副作用
```

某些项目只有其中部分环节，或存在异步、插件、生成、声明式和跨仓库关系。按真实实现记录，不强行补齐预设层次。

每个代码事实至少记录：

- 文件、资源或其他可访问定位
- 符号、配置键、声明项或可定位片段
- 当前职责和实际关系
- 与需求的关系
- 证据边界

需求文档中的路径和代码片段不是证据，必须回到本地权威实现来源验证。

## 7. 代码版本边界

记录当前可获得的：

- 仓库或代码单元
- 分支
- commit、tag、版本、文件状态或等价快照标识
- 无版本控制时的分析时点和必要文件状态

若代码在主分析与独立复核之间变化，重新核验相关证据。代码快照发生变化后，旧 `PASS` 自动失效。
