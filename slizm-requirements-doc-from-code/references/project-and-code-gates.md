## 1. 识别候选代码单元

目标可能是单体、多模块、多仓库、公共库、命令行工具、桌面或移动应用、插件、数据流程、声明式配置、基础设施定义或其他形态。

根据实际存在的内容检查：

- 项目、产品、组件、仓库、模块、包和命名空间；
- 项目说明、构建与依赖清单、入口、公开契约、配置和资源；
- 需求中的专有术语、标识符、路径、命令、协议和数据结构；
- 输入输出、状态变化、持久化、生成过程和外部依赖；
- 测试、运行、部署、发布描述和跨代码单元关系。

工作区包含多个独立代码单元时分别列出候选项。

## 2. 项目一致性

分别记录：

- **用户指定范围**：用户要求分析的位置。
- **代码身份核验**：本地证据表明该位置实际是什么。
- **一致性结论**：需求与本地代码是否对应。

### 一致

一个决定性证据或多个相互独立的强佐证建立对应关系，且没有反向证据。有效证据包括唯一标识、公开契约、实际生效路径、项目元数据，以及多个专有对象和规则的共同吻合。

目录名相似、单个通用词语相同、待审文档中的路径声明或 `.codegraph/` 存在本身不是充分证据。

### 不一致

项目用途、业务域、公开契约、核心标识或可观察行为存在明确冲突。

### 无法确认

存在多个同样合理的候选项，或证据不足以唯一定位。

结论不是“一致”时停止代码深挖和成文，说明用户指定范围、需求指向、本地识别、不匹配或缺失证据，以及需要补充的代码位置。

## 3. CodeGraph 项目边界

先确认目标业务项目根目录，再检查该根目录的 `.codegraph/`。Skill 仓库、父工作区或其他仓库的索引不能证明目标业务项目。

CodeGraph 查询返回的专有符号、入口和关系可以辅助项目身份核验，但必须与构建文件、包结构、项目元数据、公开契约、远程信息或需求材料共同判断。索引路径、分支或 commit 与目标代码不一致时，不使用其结果证明项目或代码事实。

项目一致后，按 [CodeGraph 代码取证协议](codegraph-integration.md) 选择 MCP、CLI 或普通取证路径。没有索引或能力不可用不改变项目身份结论，只改变证据方式和披露范围。

## 4. 从需求推导必需实现事实

项目一致后，逐条确定为了判断当前行为必须找到：

- 触发或使用方式；
- 负责关键判断、转换、计算、编排或状态变化的实现或声明；
- 数据、状态、配置、资源或依赖来源；
- 输出、保存、传递、生成或副作用；
- 能证明当前行为的测试、契约、配置、部署定义或生成规则。

只检查与原始需求事实判断直接相关的内容。

## 5. 权威实现来源

优先读取实际生效的来源：

- 运行时代码和调用关系；
- 声明式配置和基础设施定义；
- schema、契约、规则文件和生成来源；
- 构建、部署或运行环境消费的配置；
- 数据转换、模型和工作流定义；
- 证明当前行为的测试和固定样例。

CodeGraph 用于发现这些来源和关系。返回完整当前源码时可以作为读取证据；只返回摘要或关系时继续读取源文件。README、注释、历史设计和示例属于辅助证据。与实际生效来源冲突时记录冲突，不静默选边，也不由此扩大需求。

## 6. 代码充分性

### 完整

能够确认需求涉及的全部现状结论、核心规则、事实来源、执行关系和必要边界。

### 部分完整但足以成文

只有同时满足以下条件才能使用：

1. 每个代码现状结论都有实际证据。
2. 缺失内容不会改变明确需求的业务含义。
3. 缺失内容不会改变规范性验收标准。
4. 文档不会把未分析部分伪装成完整影响范围。
5. 缺口及 CodeGraph 降级或新鲜度限制已披露。

### 关键实现缺失

缺失会使当前规则、可观察行为、事实来源、需求与现状关系、需求相关边界或验收含义只能猜测。

关键实现缺失时停止成文，说明已确认事实、缺失事实、缺失影响和需要用户提供的位置。

## 7. 分析与引用

按真实项目形态建立证据路径：

```text
需求项
→ 触发或使用方式
→ 关键实现或声明
→ 状态、数据、配置、资源或依赖
→ 可观察结果或副作用
```

每个代码事实至少记录 CodeGraph 路由状态、语义路径、文件或资源定位、符号或配置项、当前职责、实际关系、需求关系和证据边界。需求材料、CodeGraph 或待审文档中的路径和代码片段必须回到当前权威来源验证。

## 8. 版本边界

记录仓库或代码单元、分支、commit、tag、版本、文件状态、索引状态或分析时点中的可用项。交付前相关代码发生变化时，重新核验受影响证据。索引与源码冲突时以直接读取为准并标记过期。
