本协议确认需求是否指向当前代码，以及可用实现是否足以支撑准确成文。按项目真实形态分析，不套用固定架构。

## 1. 识别候选代码单元

目标可能是单体、多模块、多仓库、公共库、命令行工具、桌面或移动应用、插件、数据流程、声明式配置、基础设施定义或其他形态。

根据实际存在的内容检查：

- 项目、产品、组件、仓库、模块、包和命名空间；
- 项目说明、构建与依赖清单、入口、公开契约、配置和资源；
- 需求中的专有术语、标识符、路径、命令、协议和数据结构；
- 输入输出、状态变化、持久化、生成过程和外部依赖；
- 测试、运行、部署、发布描述和跨代码单元关系。

工作区包含多个独立代码单元时分别列出候选项。

## 2. 项目一致性

分别记录：

- **用户指定范围**：用户要求分析的位置。
- **代码身份核验**：本地证据表明该位置实际是什么。
- **一致性结论**：需求与本地代码是否对应。

### 一致

一个决定性证据或多个相互独立的强佐证建立对应关系，且没有反向证据。有效证据包括唯一标识、公开契约、实际生效路径、项目元数据，以及多个专有对象和规则的共同吻合。

目录名相似、单个通用词语相同或待审文档中的路径声明不是充分证据。

### 不一致

项目用途、业务域、公开契约、核心标识或可观察行为存在明确冲突。

### 无法确认

存在多个同样合理的候选项，或证据不足以唯一定位。

结论不是“一致”时停止代码深挖和成文，说明用户指定范围、需求指向、本地识别、不匹配或缺失证据，以及需要补充的代码位置。

## 3. 从需求推导必需实现事实

项目一致后，逐条确定为了判断当前行为必须找到：

- 触发或使用方式；
- 负责关键判断、转换、计算、编排或状态变化的实现或声明；
- 数据、状态、配置、资源或依赖来源；
- 输出、保存、传递、生成或副作用；
- 能证明当前行为的测试、契约、配置、部署定义或生成规则。

只检查与原始需求事实判断直接相关的内容。

## 4. 权威实现来源

优先读取实际生效的来源：

- 运行时代码和调用关系；
- 声明式配置和基础设施定义；
- schema、契约、规则文件和生成来源；
- 构建、部署或运行环境消费的配置；
- 数据转换、模型和工作流定义；
- 证明当前行为的测试和固定样例。

README、注释、历史设计和示例属于辅助证据。与实际生效来源冲突时记录冲突，不静默选边，也不由此扩大需求。

## 5. 代码充分性

### 完整

能够确认需求涉及的全部现状结论、核心规则、事实来源、执行关系和必要边界。

### 部分完整但足以成文

只有同时满足以下条件才能使用：

1. 每个代码现状结论都有实际证据。
2. 缺失内容不会改变明确需求的业务含义。
3. 缺失内容不会改变规范性验收标准。
4. 文档不会把未分析部分伪装成完整影响范围。
5. 缺口已作为证据边界披露。

### 关键实现缺失

缺失会使当前规则、可观察行为、事实来源、需求与现状关系、需求相关边界或验收含义只能猜测。

关键实现缺失时停止成文，说明已确认事实、缺失事实、缺失影响和需要用户提供的位置。

## 6. 分析与引用

按真实项目形态建立证据路径：

```text
需求项
→ 触发或使用方式
→ 关键实现或声明
→ 状态、数据、配置、资源或依赖
→ 可观察结果或副作用
```

每个代码事实至少记录文件或资源定位、符号或配置项、当前职责、实际关系、与需求的关系和证据边界。需求材料或待审文档中的路径和代码片段必须回到本地权威来源验证。

## 7. 版本边界

记录仓库或代码单元、分支、commit、tag、版本、文件状态或分析时点中的可用项。主分析与独立复核之间代码发生变化时，重新核验相关证据，旧 `PASS` 失效。
