按实际需求裁剪。本文档止于业务目标、当前事实、需求规则和验收结果，不包含技术设计、预计修改文件、实施顺序或代码方案。

## 1. 内容边界与编号

### 四类信息

- **需求事实**：目标、行为、业务规则、非功能要求、验收和明确范围，只来自有效需求材料。
- **代码事实**：当前触发、规则、状态、数据、配置、资源、依赖和执行关系，由本地实际生效来源证明。
- **文档整理**：结构化和澄清表达，不改变需求含义或强度。
- **待确认项**：现有材料不能唯一决定的内容。

代码事实不能自动成为目标或保持不变项。只有有效需求明确要求保持某项行为时，才能把它写成规范性保持不变项或验收门槛。

### 编号

- 已有稳定需求编号原样保留，不改前缀、不重排、不复用。
- 仅给没有编号的规范性陈述分配未占用的 `REQ-nnn`。
- 每个 `REQ` 引用对应 `SRC-nnn` 和原文位置；同一需求有多个来源时全部记录。
- 验收项映射到需求编号；代码现状和文档整理不占用 `REQ`。

### 可选审计元数据

文档 ID、版本或修订号、已有外部审查记录和基线摘要是可选审计元数据，不是下游交接头，也不是本 skill 的完成条件。存在时保留并核验；缺失时不补造历史信息、不阻断成文，也不自动启动审查。

## 3. 按项目形态裁剪

使用项目的真实术语组织内容。公共库围绕公开使用方式和输入输出；命令行工具围绕命令、参数、输出与退出行为；配置或基础设施项目围绕声明、变量、资源与生效条件；数据流程围绕输入、转换、状态与输出；交互应用围绕操作、状态与反馈。

这些是裁剪示例，不是固定分类。省略没有事实内容的章节。

## 4. 验收标准规则

验收标准必须：

- 只验证有效需求和明确要求保持不变的行为；
- 使用可观察、可判断的结果；
- 覆盖名称、文案、对象、属性、数值、单位、条件、顺序、时间、否定和例外；
- 映射需求编号和来源，并与需求明细及追溯矩阵一致。

不得仅因当前代码存在某行为就要求它继续保持，也不得擅自加入错误恢复、重试、限流、日志、监控、权限、安全、兼容性、性能或迁移要求。会改变业务含义而来源未定义的边界行为转为 `NEEDS_INPUT`。

## 5. 追溯检查

成文前确认完整链路：

```text
SRC → REQ → 验收项 → 代码现状
```

每个规范性陈述和验收项都有需求来源；每个现状结论都有实际读取的代码证据；已有需求编号被保留；文档整理没有改变需求强度；关联代码没有被写成拟修改范围；模板没有引入技术设计字段。

最终执行结果格式以主 `SKILL.md` 为单一来源。
