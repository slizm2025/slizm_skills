按原始需求实际范围裁剪。本文档止于业务目标、当前事实、需求规则和验收结果，不输出技术设计、预计修改文件、实施顺序或代码方案。

## 1. 需求与代码的写入边界

### 规范性内容

目标要求、新增/修改/删除/保持不变行为、业务规则、非功能要求、验收标准和明确范围只能来自有效原始需求。每一项引用 `SRC-nnn` 及原句、文档位置或图片区域。

### 代码现状内容

当前触发方式、规则、可观察行为、状态、数据、配置、资源、依赖、执行关系和关联代码范围由本地权威实现来源证明。代码现状不能自动成为目标要求。

### 保持不变与回归

只有用户明确要求某行为保持不变时，才能把它写成规范性保持不变项或验收门槛。代码分析发现但用户未要求的依赖现状，可以写入当前现状或证据边界。

### 载荷与复核元数据

文档分为两部分：

- **被复核载荷**：除“文档说明”中的复核状态字段和“独立复核结果”章节之外的全部内容。
- **复核证明元数据**：复核状态字段和“独立复核结果”章节，只能在复核完成后据实写入或更新。

只有被复核载荷变化才使既有 `PASS` 失效；写入或更新复核证明元数据不触发失效。复核绑定必须记录被复核载荷的版本或内容摘要，使任何载荷改动可被发现。

## 3. 按项目形态裁剪

使用当前项目的真实术语组织内容。例如：

- 公共库围绕公开使用方式、输入、行为和输出；
- 命令行工具围绕命令、参数、规则、输出和退出行为；
- 配置或基础设施项目围绕声明、变量、资源结果和生效条件；
- 数据流程围绕输入、转换规则、状态和输出；
- 交互应用围绕用户操作、状态和反馈；
- Web 或服务系统只在需求和代码涉及时记录交互、契约和数据。

这些是裁剪示例，不是固定分类。没有事实内容的章节可以省略。

## 4. 验收标准规则

验收标准必须：

- 只验证有效原始需求和用户明确要求保持不变的行为；
- 使用可观察、可判断的结果；
- 覆盖原文中的名称、文案、对象、属性、数值、单位、条件、顺序、时间和交互；
- 每项可追溯到有效来源，并与需求明细和追溯矩阵一致。

验收标准不能仅因当前代码存在某行为就要求它继续保持，也不能擅自加入错误恢复、重试、限流、日志、监控、权限、安全、兼容性、性能或迁移要求。

原始需求没有定义且会改变业务含义的边界行为，需要用户确认，不能用通用最佳实践填空。

## 5. 追溯检查

成文前确认：

- 目标和验收项只引用有效原始需求；
- 当前现状只引用实际读取的权威实现来源；
- 规范性保持不变项同时有用户依据和当前代码证据；
- 文档整理没有改变需求强度；
- 没有遗漏数字、单位、阈值、顺序、否定词、例外和限定范围；
- 关联代码范围没有被写成拟修改范围。

最终执行结果格式以主 `SKILL.md` 的“最终回复”为单一来源。
