本协议确保需求结论都能回到一手材料，而不是依赖作者摘要。

## 1. 来源范围

纳入所有会建立、修改、撤回或解释需求范围的用户材料：

- 文档、表格、PDF 和其他附件；
- 原图、截图、扫描件和原型；
- 用户在对话中直接输入的文字；
- 后续补充、更正、替换、撤回和澄清；
- 用户明确指定为需求依据的其他资源。

作者摘要、推理、建议和代码结论不是原始需求来源。只登记当前环境实际可见或用户明确提供的内容。

## 2. 来源登记

为每个独立来源分配稳定编号 `SRC-001`、`SRC-002`……，保留来源边界，不先合并成“最终理解”。

| 字段 | 说明 |
|---|---|
| 来源编号 | 稳定的 `SRC-nnn` |
| 类型 | 文档、图片、对话原文、附件或其他 |
| 定位符 | 文件路径、附件 ID、URI、资源句柄或完整正文 |
| 顺序 | 用户提供或确认的顺序 |
| 状态 | 有效状态 |
| 关系 | 与其他来源的补充、替换、撤回或冲突关系 |
| 关系依据 | 用户明确建立关系的原话或位置 |
| 完整性标识 | 可选哈希、版本、修改时间或内容摘要 |

定位符应让当前执行者能够重新访问和核验来源。

### 对话或只读模式

- 文件和附件提供可访问定位符。
- 对话文字按消息分别保留逐字内容和顺序。
- 不为流程方便额外创建证据包。

### 长期审计模式

用户要求落盘时可以建立 manifest 或证据目录。使用用户指定或明确披露的位置，先检查目标，避免覆盖未知文件。完整性标识用于发现变化，不替代内容核验。

## 3. 状态与关系

来源状态：

- `active`：当前有效。
- `withdrawn`：用户明确撤回。
- `superseded`：被用户明确指定的新来源整体替换。
- `partially-superseded`：只有指定片段或图片区域被替换。
- `unresolved-conflict`：来源冲突且用户未裁决。

关系包括 `supplements`、`replaces`、`replaces-fragment`、`withdraws`、`clarifies` 和 `conflicts-with`。

补充、替换、撤回和部分覆盖关系必须有用户明确依据。时间较晚本身不代表优先级更高。

建立关系时记录源来源、目标来源、目标片段或图片区域、关系类型和用户原话依据。冲突会改变需求含义、范围或验收结果时，将相关来源标为 `unresolved-conflict` 并询问用户。

## 4. 对话和视觉材料

### 对话文字

- 保留用户原词、数字、单位、标点、换行、列表和顺序。
- 后续更正作为新来源登记，不静默修改旧来源。
- 保留澄清、范围解释和撤回说明。
- 排除作者分析和与需求无关的闲聊。

只有改写摘要而没有用户原文时，不能把摘要标记为原始来源。

### 图片与视觉材料

当前执行者必须使用环境提供的视觉读取能力查看每张有效原图。

- OCR、转录和图片描述只能辅助。
- 多张图片分别编号，保留用户给出的顺序和关系。
- 引用时使用来源编号、文件名和可定位区域。
- 原图不可访问、分辨率不足或关键视觉关系无法判断时标记不可验证；影响核心需求时输出 `NEEDS_INPUT`。

## 5. 需求基线

从所有 `active` 来源和 `partially-superseded` 来源中的有效片段提取：

- 背景和目标；
- 新增、修改、删除和明确保持不变的行为；
- 角色、对象、触发方式和可观察结果；
- 名称、文案、属性、数值、单位、条件、顺序、时间和交互；
- 原文明确给出的边界、非功能要求和验收标准；
- 冲突、歧义和缺失信息。

将内容分类为需求事实、代码事实、文档整理和待确认项。代码事实不能成为需求事实，文档整理不能增加要求。

## 6. 版本与失效

记录来源编号清单、附件版本、文件修改时间、哈希或证据时点中的可用项。

新增、修改、撤回或重新解释需求来源，原图或附件变化，以及来源状态或关系变化，都会使旧需求基线失效。变化后重新建立来源集和需求基线。
