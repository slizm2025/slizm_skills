# 有效需求来源协议

在提取需求基线前使用本协议。目标是让主代理和 clean-room 复核者能够分别回到一手材料，而不是共同依赖主代理的摘要。

## 1. 来源范围

把所有会建立、修改、撤回或解释需求范围的用户材料纳入来源集，包括：

- 文档、表格、PDF 或其他附件
- 原图、截图、扫描件、原型图
- 用户在对话中直接输入的文字
- 用户后续给出的补充、更正、替换、撤回和澄清
- 用户明确指定为需求依据的其他资源

主代理的摘要、推理、建议和代码结论不是原始需求来源。

只记录当前环境实际可见或用户明确提供的内容。不要声称掌握平台未暴露的消息编辑历史；若不可见的旧版本会影响需求，需要用户补充。

## 2. 来源登记

为每个独立来源分配稳定编号 `SRC-001`、`SRC-002`……。保留来源边界，不先合并成“最终理解”。

来源登记至少包含：

| 字段 | 说明 |
|---|---|
| 来源编号 | 稳定的 `SRC-nnn` |
| 类型 | 文档、图片、对话原文、附件或其他 |
| 定位符 | 文件路径、附件 ID、URI、资源句柄或完整正文 |
| 顺序 | 用户提供或确认的顺序 |
| 状态 | 来源有效状态 |
| 关系 | 与其他来源的补充、替换、作废或冲突关系 |
| 关系依据 | 用户明确建立该关系的原话或位置 |
| 完整性标识 | 可选的哈希、版本、修改时间或内容摘要 |

定位符只需让当前主代理和新的复核代理都能访问，不要求必须是绝对路径，也不要求创建固定临时目录。

### 两种载体模式

**仅对话/只读模式**：

- 本地文件或附件提供可访问定位符。
- 对话文字按消息分别逐字提供给复核者。
- 待审文档以完整正文传递。
- 不创建证据包文件。

**落盘/长期审计模式**：

- 可建立 manifest 或证据目录，但应使用用户指定或明确披露的位置。
- 不直接覆盖未知证据文件。
- 完整性标识用于发现材料变化，不证明最初抄录一定正确。

## 3. 状态与关系

来源状态：

- `active`：当前有效。
- `withdrawn`：用户明确撤回，不再作为目标依据，但保留审计记录。
- `superseded`：被用户明确指定的新来源整体替换。
- `partially-superseded`：只有特定片段或图片区域被替换，其余部分仍有效。
- `unresolved-conflict`：来源冲突且用户未明确裁决。

常用关系：

- `supplements`：补充，不覆盖原内容。
- `replaces`：整体替换。
- `replaces-fragment`：替换指定原句、段落、表格单元或图片区域。
- `withdraws`：明确作废。
- `clarifies`：解释含义，但不增加新目标时仍保留原文依据。
- `conflicts-with`：存在矛盾且尚无裁决。

建立覆盖关系时必须记录：源来源、目标来源、目标片段或图片区域、关系类型和用户原话依据。所有补充、替换、撤回和部分覆盖关系都必须由用户明确建立，不能由主代理依据时间或语义相似度推断。

时间较晚不自动代表优先级更高。只有用户明确表达“以新版为准”“上一条作废”“把 X 改为 Y”等含义时，才建立替换或作废关系。

若冲突会改变需求含义、范围或验收结果，把相关来源标为 `unresolved-conflict`，停止成文并询问用户。

## 4. 对话文字

- 逐字保留用户原词、数字、单位、标点、换行、列表和顺序。
- 用户后续编辑或更正作为新的来源登记，不静默修改旧来源。
- 保留需求澄清、范围解释和撤回说明，即使它们出现在争论或长对话中。
- 排除主代理分析和与需求无关的闲聊。
- 如果只有改写后的摘要而没有用户原文，不能把摘要伪装成原始来源。

## 5. 图片与视觉材料

主代理和 clean-room 复核者必须分别使用当前环境支持的原始视觉读取能力直接查看每张有效原图。

- OCR、转录、图片描述和提取文字只能作为辅助。
- 不得让复核者只读取主代理生成的转录。
- 多张图片分别编号，并保留用户给出的顺序和关系。
- 引用图片证据时使用来源编号、文件/附件名和可定位区域，例如“顶部标题”“红框内第二行”“表格第 3 行”。
- 若原图不可访问、分辨率不足、内容被遮挡或关键视觉关系无法可靠判断，标记为不可验证；关键需求受影响时输出 `NEEDS_INPUT`。

## 6. 需求基线

从所有 `active` 来源以及 `partially-superseded` 来源中仍有效的片段提取需求基线：

- 背景和目标
- 明确新增、修改、删除和保持不变的行为
- 角色、对象、触发方式和可观察结果
- 名称、文案、属性、数值、单位、条件、顺序、时间和交互
- 原文明确给出的边界、非功能要求和验收标准
- 冲突、歧义和缺失信息

将内容分为：

- **需求事实**：有效来源直接表达的目标或约束。
- **代码事实**：本地权威实现来源能够证明的当前行为。
- **文档整理**：不改变含义的结构化表达。
- **待确认项**：不能由有效需求来源或代码唯一确定的内容。

代码事实不能成为需求事实；文档整理不能增加要求。

## 7. 版本与失效

记录可获得的来源集版本，例如来源编号清单、附件版本、文件修改时间、内容哈希或证据时点。完整性标识是可选辅助，不是 PASS 的必要条件。

以下任一变化会使旧需求基线和旧独立复核结论失效：

- 新增、修改、撤回或重新解释需求来源
- 原图、文档或附件内容变化
- 来源状态或覆盖关系变化

发生变化后重新建立有效来源集和需求基线，并重新执行完整复核。
