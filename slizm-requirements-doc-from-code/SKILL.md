# 基于原始需求与代码生成需求文档

以有效原始需求定义目标，以本地实际生效的实现来源证明现状，生成可追溯、可评审、可验收的需求文档。

## 完成标准

最终文档必须满足：

- 每个目标要求、保持不变项和验收标准都有有效原始需求依据。
- 每个代码现状结论都有实际读取的一手实现证据。
- 需求事实、代码事实、文档整理和待确认项没有混淆。
- 项目一致，代码完整或部分完整但足以成文。
- 需求、现状、目标变化、验收和追溯矩阵互相一致。
- 最新需求、代码和文档版本已经获得 clean-room `PASS`。

## 产物边界

- 本 skill 只负责需求分析和成文，不输出技术方案、预计修改文件、实施顺序或代码改动建议。
- 用户要求写入文件时先检查目标；未知同名文件不直接覆盖。默认文件名为 `<需求主题>-需求文档.md`。
- 用户要求只读或仅在对话输出时，不创建证据包或文档文件。
- 保持业务代码、配置、测试和其他项目文件不变。
- 用户明确要求快速分析或草稿时，可以在作者自检后输出，但必须醒目标记“草稿—独立复核未执行”，且不能使用最终完成表述。

## 强制工作流

### 1. 建立有效需求来源集

读取并执行 [有效需求来源协议](references/evidence-protocol.md)。为每个独立来源分配 `SRC-nnn`，保留来源边界、顺序、定位符、状态和用户明确建立的关系。

从全部有效来源提取需求基线，并区分：

- **需求事实**：来源直接表达的目标或约束。
- **代码事实**：本地权威实现来源证明的当前状态。
- **文档整理**：不改变含义的结构化表达。
- **待确认项**：现有材料不能唯一决定的内容。

本步骤完成条件：所有需求材料已登记；每个有效片段已进入基线或标记为冲突、撤回或被替换。

### 2. 执行项目一致性与代码充分性门禁

读取并执行 [项目一致性与代码充分性门禁](references/project-and-code-gates.md)。用户指定范围只定义分析位置，不能代替代码身份核验。

项目不一致、无法确认或关键实现缺失时，停止成文，说明已确认事实、缺失证据、受影响需求和需要用户提供的内容。

本步骤完成条件：项目一致，且每个将写入文档的代码现状结论都有足够证据。

### 3. 独立分析实际实现

围绕每条需求搜索实际生效的权威来源，不做无关全库审计。根据项目真实形态建立证据路径：

```text
需求项
→ 触发或使用方式
→ 关键实现、声明或生成来源
→ 状态、数据、配置、资源或依赖
→ 可观察结果或副作用
```

记录文件或资源定位、符号或配置项、当前职责、执行关系、与需求的关系和证据边界。记录可获得的分支、commit、版本或分析时点。

本步骤完成条件：文档中拟写入的每个当前事实都能回到已读取的一手来源。

### 4. 建立追溯并生成初稿

维护内部追溯：

| 需求编号 | 有效原始来源 | 当前代码证据 | 目标变化 | 验收依据 | 确定性 |
|---|---|---|---|---|---|

检查每个目标和验收项都来自有效需求；代码发现的现状不能自动升级为目标、保持不变项或规范性验收要求。

生成初稿前读取 [需求文档结构与验收协议](references/requirements-document-template.md)，按实际需求裁剪章节。

本步骤完成条件：每个目标和验收项都有来源，每个现状结论都有代码证据，且没有由模板制造的内容。

### 5. 作者自检

逐项检查：

- 忠实性：没有遗漏、弱化、强化或改变原始需求。
- 范围：没有加入无依据产品或技术要求。
- 事实：代码现状均有一手证据。
- 分离：需求事实、代码事实和待确认项没有混淆。
- 追溯：目标、验收和现状结论均可定位来源。
- 边界：项目、代码和不可验证范围如实披露。

### 6. clean-room 独立复核

最终版交付前读取并执行 [clean-room 独立复核协议](references/independent-review-protocol.md)。只向全新隔离审查者提供有效原始来源、工作区或代码范围、代码快照、完整待审文档和固定规则。

- `PASS`：版本绑定有效时可以交付最终版。
- `FAIL`：按证据最小修订，再由另一个全新审查者复核完整文档。
- `NEEDS_INPUT`：停止最终交付并提出聚焦问题。

需求来源、代码快照或待审文档的被复核载荷发生任何变化，旧 `PASS` 失效；复核完成后据实写入复核证明元数据（复核状态字段、独立复核结果章节）不使 `PASS` 失效。完整复核累计不超过 3 轮，超限转 `NEEDS_INPUT`。环境不能保证隔离或完整传递一手材料时，产物只能标记为草稿。

## 提问规则

只在答案会改变需求含义、来源有效性、代码范围、项目归属或交付结论时提问。优先一次列出相关缺口。

必须提问：来源冲突、关键图片不可辨认、项目无法确认、关键实现缺失、需求与现状冲突且目标含义不明，或规范性要求存在多个会改变结果的合理解释。

可以通过继续读取材料或代码确认的事实，以及不影响业务含义的排版和措辞问题，由代理自主处理。

## 最终回复

只有最新复核为 `PASS` 时使用：

```text
执行结果：
- 需求证据版本：
- 用户指定范围：
- 项目一致性：一致（PASS 必要条件；不满足时禁用本模板）
- 代码充分性：完整 / 部分完整但足以成文
- 代码证据版本：
- 动态分析范围：
- 需求文档：文件位置 / 已在对话输出
- 文档修订号与被复核载荷摘要：
- 独立复核：PASS
- 复核轮次与问题闭环：
- 阻断性待确认项：无（PASS 必要条件；存在阻断项时禁用本模板）
- 已披露证据边界：无 / 列表
- 忠实性说明：未增加原始需求之外的功能、优化或技术要求
```

草稿必须报告独立复核未执行的原因和准确性限制。不要声称读取了未访问的来源或代码。
