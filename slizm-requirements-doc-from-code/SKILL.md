# 根据需求与代码编写需求文档

以需求材料定义目标，以本地实际生效的实现来源证明现状。只产出业务需求文档，不产出技术设计、预计文件清单、实施顺序或代码改动。

## 结果状态

- `FINAL`：文档已完成来源核验、代码取证、追溯检查和作者自检，且没有阻断项。
- `DRAFT`：用户明确只要草稿，或存在不阻断继续成文但会限制准确性或完整性的证据边界。
- `NEEDS_INPUT`：来源冲突、项目不匹配、关键代码缺失或歧义会改变需求含义，现有材料不足以继续。

只有 `FINAL` 可以使用“最终完成”或等价表述。`DRAFT` 必须说明限制；`NEEDS_INPUT` 只输出已确认事实、阻断项和聚焦问题。是否另行审查不影响本 skill 的结果状态。

## 输入与边界

独立识别并组合当前可用输入，不要求任何其他 skill 的产物：

- 原始需求：对话原文、附件、图片、表格、原型或其他一手材料；
- 已有需求文档或 PRD：用于补全、更新或与代码核验；
- 多份补充材料：保留用户明确建立的补充、替换、撤回和冲突关系。

按内容语义判断权威性，不按文件名、目录或生成来源判断。已有文档 ID、版本和外部审查记录属于可选审计元数据：存在时保留并核验，缺失时不阻断本次工作，也不要求补做审查。用户要求写入文件时先检查目标；来源不明的同名文件不直接覆盖。默认文件名为 `<需求主题>-需求文档.md`。用户只要求对话输出时不创建文件。

保持业务代码、配置、测试和其他项目文件不变。

## CodeGraph 代码路由

需要理解或定位代码时，在首次代码搜索或源码读取前完整读取并执行 [CodeGraph 代码取证协议](references/codegraph-integration.md)。确认目标业务项目根目录并检查其 `.codegraph/`：存在时执行 CodeGraph-first；不存在或能力不可用时按协议记录状态并使用普通取证。CodeGraph 只证明代码现状，不是需求来源，也不替代源码核验或项目身份验证。

## 工作流

### 1. 建立需求来源集

完整读取并执行 [有效需求来源协议](references/evidence-protocol.md)。登记原始需求、已有需求文档和所有补充材料，为独立来源分配 `SRC-nnn`，保留来源边界、定位、状态、顺序和用户明确关系。

始终区分：

- **需求事实**：有效来源直接表达的目标、约束和验收；
- **代码事实**：本地权威实现来源证明的当前状态；
- **文档整理**：不改变含义的结构化表达；
- **待确认项**：现有材料不能唯一决定的内容。

本步骤完成条件：每份输入均已登记；每个有效片段已纳入基线，或标记为撤回、替换、冲突或待确认。

### 2. 保留并分配需求编号

扫描全部规范性陈述：

- 已有稳定编号原样保留，不改前缀、不重排、不复用；
- 仅给没有编号的规范性陈述分配未占用的 `REQ-nnn`；
- 同一陈述在多处出现时保留一个编号并记录全部来源；
- 验收项引用对应需求编号，不把代码现状或文档整理编号成需求。

本步骤完成条件：每个规范性陈述都有稳定编号，或因冲突、歧义被登记为阻断项。

### 3. 核验项目与代码充分性

读取并执行 [项目一致性与代码充分性门禁](references/project-and-code-gates.md)。用户指定路径只定义分析位置，不能代替项目身份核验。CodeGraph 路由只使用目标业务项目根目录的索引，并记录路由状态、分支或 commit 与索引新鲜度。

项目不一致、无法确认或关键实现缺失时转为 `NEEDS_INPUT`。代码部分完整但足以准确描述已验证现状时，披露证据边界后继续。

本步骤完成条件：项目一致；CodeGraph 路由已完成或明确不适用；每个拟写入的代码现状结论都有足够的一手实现证据。

### 4. 沿实际生效路径分析现状

按需求主题执行整体语义查询，定位入口、关键实现、接口与实现、状态或数据来源、配置、依赖、输出、副作用和现有测试。结果不足时再查询调用者、动态实现、字段写入、配置生效路径和测试覆盖。CodeGraph 未覆盖或结果不完整的范围使用直接读取和普通搜索补充。

对每个拟写入文档的代码事实记录：

| 字段 | 内容 |
|---|---|
| CodeGraph 状态 | MCP、CLI、未索引、失败或过期降级 |
| 语义路径 | 入口到可观察结果的实际关系 |
| 一手定位 | 文件、符号或配置项及行号 |
| 证据方式 | CodeGraph 返回的完整源码或直接读取 |
| 新鲜度限制 | 无、局部过期或整体过期 |

代码现状不得自动形成目标、保持不变项或验收标准。现有行为是否应保留会改变需求含义时，把它登记为待确认项。

本步骤完成条件：文档中的每个当前事实都能回到当前源码或其他实际生效来源；语义关系、新鲜度与降级范围均已记录。

### 5. 建立追溯并生成文档

读取 [需求文档结构与验收协议](references/requirements-document-template.md)，按事实裁剪章节。维护以下追溯，不增加技术设计字段：

```text
SRC → REQ → 验收项 → 代码现状
```

内部代码证据同时保留 CodeGraph 状态、语义路径、一手定位、证据方式和新鲜度限制。代码发现只能证明现状，不能自动升级为规范性内容。

本步骤完成条件：每个 `REQ` 和验收项都有有效需求来源，每个代码现状都有一手实现证据，且模板和 CodeGraph 都没有制造新要求。

### 6. 作者自检

检查每个编号、数字、单位、条件、顺序、否定词、例外和限定范围；确认没有遗漏、弱化、强化或技术化改写需求。确认应调用时已 CodeGraph-first，语义结果已核验到当前源码，未索引、失败或过期范围已披露，代码行为没有被升级成目标要求。

本步骤完成条件：所有规范性陈述、验收项、代码事实和 CodeGraph 路由状态均已逐项核对，未解决的实质问题已转为 `NEEDS_INPUT`。

## 审查边界

本 skill 在作者自检和交付后结束。不得把启动子 agent、独立审查实例或审查修订循环作为完成步骤，也不得因为没有审查实例而降级结果。

是否审查由使用者在交付后自行决定。需要审查时，在新的 agent 或 task 中单独发起，并按审查目标提供最终文档、必要的一手需求材料和代码范围；审查过程及结论不由本 skill 自动创建、等待或闭环。

## 提问规则

只在答案会改变需求含义、来源有效性、项目归属、代码范围或结果状态时提问。优先一次列出相关缺口。可以通过继续读取材料或代码确认的事实，以及不影响业务含义的排版和措辞，由代理自主处理。

```text
- 保留及新增的需求编号：
- 用户指定范围：
- 项目一致性：
- 代码充分性与代码基线：
- CodeGraph：路由状态、目标索引根目录与新鲜度限制
- 需求文档：文件位置 / 已在对话输出
- 后续审查：未自动执行；是否另开 agent 由使用者决定

```
