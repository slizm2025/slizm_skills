# 基于原始需求与代码生成需求文档

将有效原始需求作为目标范围的唯一依据，将本地代码作为当前事实的证据，生成可追溯、可评审、可验收的需求文档。

本 skill 只负责需求分析和成文，不负责技术设计、预计修改文件清单或代码实现。

## 核心原则

1. **需求定义目标**：目标要求、保持不变项和验收标准只能来自有效原始需求。
2. **代码证明现状**：代码用于确认当前行为、术语、数据或状态来源、执行关系和关联范围，不能自动升级为新增需求。
3. **保持忠实**：不得补充原始需求未要求的功能、优化、重构、异常策略、兼容策略、权限、安全、性能、监控、重试或其他工程要求。
4. **证据优先**：需求结论引用原始来源；代码现状引用实际读取的权威实现来源。摘要、注释、历史文档和待审文档中的代码片段不能替代一手证据。
5. **不确定即停**：来源冲突、项目归属、关键实现或需求含义无法唯一确定且会影响文档时，先询问用户，不输出貌似完整的最终文档。
6. **独立复核**：最终文档必须通过没有主代理既有分析记忆的 clean-room 复核；未获得最新一轮 `PASS` 时只能称为草稿。

## 产物与写入边界

- 用户要求写入文件时，先检查目标；未知同名文件不得直接覆盖。默认文件名为 `<需求主题>-需求文档.md`。
- 用户要求只读或仅在对话输出时，不创建证据包或文档文件。将可访问的原始来源定位符和完整待审正文直接提供给复核者。
- 不修改业务代码、配置、测试或其他项目文件。
- 不生成技术方案、拟修改文件、实施顺序或代码改动建议；这些属于后续技术设计阶段。
- 独立复核者同样不得修改文件或运行有副作用的操作。

## 输入

优先获得：

- 原始需求材料：文档、图片/截图、用户原文、后续补充、更正、撤回或其他附件。
- 用户指定的项目、仓库、模块、组件、分支或代码目录。
- 输出方式：仅对话或写入指定文件。

用户未指定代码目录时，先检查当前工作区并尝试定位，不要求用户重复提供可由代码确认的信息。

## 强制工作流

### 1. 建立有效需求来源集

在提取需求前，读取 `references/evidence-protocol.md` 并按其规则执行。

- 为每个独立来源分配 `SRC-nnn`，保留来源边界、顺序、可访问定位符和用户明确声明的来源关系。
- 用户在对话中建立、修改、撤回或解释需求范围的原话属于需求证据；主代理的总结和推理不属于。
- 图片必须直接查看原图；OCR、转录或图片描述只能辅助。
- 较晚来源不自动覆盖较早来源。没有明确覆盖关系的冲突会影响需求含义时，停止并询问用户。
- 基于全部有效来源建立需求基线，并将内容区分为：**需求事实、代码事实、文档整理、待确认项**。

### 2. 执行项目一致性门禁

读取 `references/project-and-code-gates.md`，独立识别需求指向的目标和工作区候选代码单元。

判定为：

- **一致**：一个决定性证据，或多个相互独立的强佐证，将需求与代码对应起来，且没有反向证据。
- **不一致**：项目、用途、业务域或关键标识明显冲突。
- **无法确认**：存在多个合理候选项或证据不足。

“不一致”或“无法确认”时，停止代码深挖和文档成文，说明证据并询问用户。用户指定范围只说明“分析哪里”，不能覆盖明显的身份冲突。

### 3. 执行代码充分性门禁

从原始需求反推必须确认的实现事实，再按项目真实形态检查代码；不预设任何固定分层、技术栈或组件类型。

判定为：

- **完整**：足以确认所有写入文档的需求相关现状和边界。
- **部分完整但足以成文**：每项现状都有证据；缺口不改变需求含义和验收标准；文档不声称影响范围完整；缺口已明确披露。
- **关键实现缺失**：缺口会使规则、现状、来源或边界只能猜测。

“关键实现缺失”时停止并精确说明已找到内容、缺失事实和缺失影响，请用户补充对应代码。

### 4. 独立分析实际实现

围绕每条需求搜索实际生效的权威实现来源，不做无关全库审计。按项目真实形态建立证据路径：

```text
需求项
→ 触发或使用方式
→ 关键实现或声明
→ 相关状态、数据、配置、资源或依赖
→ 可观察结果或副作用
```

记录文件或资源定位、符号/配置项、执行关系、当前规则和证据边界。若代码、配置、项目说明或历史文档冲突，明确记录，不静默选边。

记录可获得的分支、commit、版本或等价代码快照标识；无法获得时明确写明证据时点。

### 5. 建立追溯并生成初稿

每条目标要求建立映射：

| 需求编号 | 有效原始来源 | 当前代码证据 | 目标变化 | 验收依据 | 确定性 |
|---|---|---|---|---|---|

检查：

- 每个目标要求和验收项都能追溯到有效原始需求。
- 每个当前实现结论都能追溯到实际读取的实现证据。
- 代码发现的现状没有被写成本次目标或规范性验收要求。
- 原始需求中的名称、文案、数值、单位、条件、顺序、时间、交互和保持不变项没有遗漏或改变。

生成初稿前读取 `references/requirements-document-template.md`。按需求实际范围裁剪章节，不为套模板编造页面、接口、字段、数据库或其他不存在的内容。

### 6. 主代理自查

在独立复核前检查：

- 忠实性：没有遗漏、弱化、强化或改变原始需求。
- 范围：没有加入无依据的产品或技术要求。
- 事实：代码现状均有一手证据。
- 分离：需求事实、代码事实和待确认项没有混淆。
- 追溯：每条目标要求、验收项和现状结论均可定位来源。
- 边界：项目、代码和不可验证范围已如实披露。

### 7. clean-room 独立复核与交付门禁

初稿完成后读取并严格执行 `references/independent-review-protocol.md`。

- 启动一个确实隔离上下文的全新审计代理。
- 只提供有效原始来源、工作区或代码范围、代码快照信息、完整待审文档，以及固定审查规则。
- 不提供主代理的需求基线、项目结论、代码结论、自查结果、搜索笔记或期望 verdict。
- 复核者必须从头读取原始来源、直接查看原图、独立识别项目并重新核验实际代码。

按 verdict 处理：

- **PASS**：只有版本绑定仍有效时才能交付。
- **FAIL**：现有材料足够但文档有可修正问题；按证据最小修订，再由另一个全新的 clean-room 代理复核完整文档。
- **NEEDS_INPUT**：必须由用户补充或裁决；停止交付并提出聚焦问题。

任何需求来源、代码快照或待审文档发生变化，旧 `PASS` 自动失效。修改后不得复用原审查代理，也不得只审查差异。

如果当前环境无法保证上下文隔离或无法让复核者访问一手材料，明确说明独立复核未完成，产物只能标记为草稿。

## 提问规则

只在答案会改变需求含义、来源有效性、代码范围、项目归属或交付结论时提问。优先一次列出相关缺口。

必须提问：

- 有效来源之间存在未解决冲突。
- 原图或关键附件不可访问或无法可靠辨认。
- 项目不一致或无法确认。
- 关键实现缺失。
- 需求与当前实现冲突且原文不能确定目标含义。
- 某条规范性要求存在多个合理解释。

不应提问：

- 可以通过继续读取一手材料或代码确认。
- 只是排版、章节名或默认文件名偏好。
- 不影响业务含义的轻微措辞。

## 最终回复

只有获得最新 clean-room `PASS` 后，才使用：

```text
执行结果：
- 需求证据版本：<来源集标识或来源数量/时点>
- 用户指定范围：<未指定或用户提供的代码范围>
- 项目一致性：一致
- 代码充分性：完整 / 部分完整但足以成文
- 代码证据版本：<branch/commit/version/时点>
- 动态分析范围：<本次实际核验的代码单元与行为>
- 需求文档：<文件位置或“已在对话中输出”>
- 独立复核：PASS
- 复核轮次：<次数>
- 问题闭环：<已修正与证据驳回数量>
- 阻断性待确认项：无
- 已披露证据边界：无 / <列表>
- 忠实性说明：未增加原始需求之外的功能、优化或技术要求
```

不要声称读取了未访问的来源或代码，不要把用户指定范围等同于项目一致性，也不要在独立 `PASS` 前声称最终准确完成。
