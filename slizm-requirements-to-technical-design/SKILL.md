# 基于需求与代码生成技术设计

以需求定义目标，以当前实际生效的代码定义落地约束。输出能够指导实施、追溯到需求和代码证据、并明确验证落点的技术设计。本 skill 可独立启动：输入既可以是上游 skill 生成的需求文档，也可以是客户或用户直接提供的原始需求方案。

## 完成标准

只有同时满足以下条件，技术设计才可标记为可实施版本：

- 每条规范性需求都有稳定编号和设计落点。
- 每个现状结论、复用点和拟修改文件都有本地权威实现证据。
- 每个技术变化同时有需求依据和代码依据。
- 文件清单、调用关系、数据变化、验证策略和实施顺序相互一致。
- 项目一致性成立，代码达到完整或部分完整但足以设计。
- 不存在会改变业务效果、公开契约、数据或模块职责的未决问题。
- 独立复核状态与当前需求、代码和文档版本一致；未执行或受上游状态传染时明确标记受限版本。

## 产物与修改边界

- 默认在对话中输出并保持业务代码、配置、测试和版本控制状态不变。
- 用户明确要求写入文件时，只创建或更新指定技术文档；未指定文件名时默认使用 `<需求主题>-技术设计.md`。先检查目标，避免覆盖来源不明的同名文件。
- 用户同时明确要求设计和实现时，先完成设计门禁；无阻塞后再进入代码实施流程。设计请求本身不构成修改业务代码的授权。
- 只读分析优先使用不会改变项目的检查。需要运行可能产生构建物、缓存或其他副作用的验证时，遵循当前环境权限和用户授权，并披露结果边界。

## 强制工作流

### 1. 固化需求基线

完整读取所有需求材料（文档、附件、图片或用户在对话中直接给出的原文）和用户明确用于补充、修正或撤回需求的原文；包含图片时使用视觉读取能力查看原图。记录来源路径或对话位置、可用版本标识、提供顺序和明确关系。

按需求材料类型确定复核状态处理：

- **没有上游交接头的材料**（客户原始需求方案、普通需求文档、附件或对话原文等）：本身就是本阶段的一手需求来源，直接作为需求基线，不适用状态传染；如实记录“无上游复核状态”即可，不要求先经过上游需求 skill。
- **带上游交接头的需求文档**（交接头含文档 ID、修订号、复核状态、有效需求来源与代码证据版本即需求/代码基线、被复核载荷摘要、已披露证据边界中的可用项）：记录交接头。上游文档标记为草稿或“独立复核未执行”时，该状态向下传染，本阶段产物最高只能标记为受限版本。解除传染的方式有两种：按上游 skill 定义的完整门禁（含 clean-room 需求复核）重新闭环上游文档，并将升级结果声明为对上游文档的复核；或用户明确声明以该文档当前内容为需求依据，此时按一手需求材料处理，并在证据边界记录该声明。本阶段自身的设计复核不能替代上游门禁。

提取并编号规范性需求：

- 需求文档已包含稳定编号（如 `REQ-nnn`）时必须原样继承，不得重新编号；只对文档中没有编号的规范性陈述补充新编号，并标注其在文档中的位置。
- 技术推导和设计决定使用独立前缀 `DES-001`、`DES-002`……，逐项标注派生自哪些 REQ，不占用 REQ 命名空间。

需求编号需覆盖：

- 新增、修改、删除和保持不变的行为；
- 角色、对象、入口、流程、状态、字段、规则、文案、数值、单位和时序；
- 明确的非功能要求与验收标准；
- 冲突、歧义和缺失信息。

始终区分：

- **需求事实**：需求来源直接表达的目标和约束。
- **代码事实**：本地权威实现来源证明的当前状态。
- **技术推导**：为在当前项目中满足需求而设计的变化。
- **待确认项**：现有材料不能唯一决定且会影响设计的内容。

本步骤完成条件：每个规范性陈述已映射到需求编号，或已登记为阻断项。

### 2. 核验项目与代码充分性

读取并执行 [项目一致性与代码充分性门禁](references/project-and-code-gates.md)。

上游需求文档记录的代码基线与当前代码不一致时，标记“基线过期”，对需求触及的代码范围重新核验；只有相关代码确实变化、或变化影响无法判定时才阻断，不因无关提交阻断。基线过期及其处理结果写入证据边界。

结论不是“项目一致，代码完整或部分完整但足以设计”时，停止形成可实施方案，向用户说明证据、缺口和影响。

本步骤完成条件：每条需求都有足以选择技术落点的现状证据，或已有精确阻断说明。

### 3. 沿实际生效路径分析现状

围绕需求逐项搜索，不做无关全库审计。根据项目真实形态建立证据路径，例如：

```text
需求项
→ 触发或使用入口
→ 关键实现、声明或生成来源
→ 状态、数据、配置、资源或外部依赖
→ 当前输出与副作用
→ 现有验证方式
```

记录文件或资源定位、符号或配置项、调用和数据关系、当前职责、与需求的关系及证据边界。实际生效的运行时代码、声明式配置、schema、生成来源、基础设施定义和测试都可能是权威证据。

本步骤完成条件：文档将引用的每个当前事实都能回到已读取的一手实现来源。

### 4. 建立需求—现状—设计追溯

维护内部矩阵：

| 需求编号 | 原始依据 | 当前代码证据 | 技术变化 | 预计文件/位置 | 验证落点 | 确定性 |
|---|---|---|---|---|---|---|

设计满足需求所需的最小变更集：

- 优先沿用现有模块、接口、类型、组件和依赖方向。
- 对每项变化说明输入、处理、输出、副作用和保持不变项。
- 拟新增路径必须由现有目录和同类命名惯例支撑。
- 需求未授权且会改变公开契约、持久化、兼容性、依赖或跨模块职责的变化列为阻断项。
- 现有唯一惯例能够解决的非实质选择由代理自主决定。

本步骤完成条件：每条需求至少有一个设计落点，每个设计变化至少映射一条需求，且没有无依据文件或技术要求。

### 5. 生成技术设计

读取 [技术设计结构与输出协议](references/technical-design-template.md)，按实际复杂度选择轻量或标准结构。章节服务于事实，不为模板补空内容。

用户要求写入文件时，先检查目标文件；若目标已存在且来源或覆盖意图不明确，先询问用户。否则在指定位置写入完整技术设计。

本步骤完成条件：需求、代码证据、整体实现、文件清单、验证和风险在文档中互相闭合。

### 6. 作者自检

逐项检查：

- 需求覆盖：没有遗漏、弱化、强化或改写需求。
- 代码真实性：现状、路径和符号均已实际读取。
- 现状/目标分离：拟新增和拟修改内容标记清楚。
- 最小变化：没有夹带无依据重构、优化或产品规则。
- 文件一致性：详细设计、文件清单、实施顺序和验证映射一致。
- 基线有效：需求和相关代码没有在分析期间变化。

### 7. 独立复核与交付门禁

读取并执行 [独立技术设计复核协议](references/independent-review-protocol.md)。使用能够建立全新隔离上下文的审查能力时，只向审查者提供原始需求来源、项目路径与代码快照、完整待审文档和固定规则；不提供作者提取的需求基线。

- `PASS`：当前版本可以交付。
- `FAIL`：根据证据修订；被复核载荷的任何修改都会使旧结论失效（据实写入复核证明元数据除外），使用新的隔离审查实例复核完整文档。
- `NEEDS_INPUT`：停止交付可实施方案并向用户提出聚焦问题。
- 无隔离审查能力：输出明确标记“独立复核未执行”的受限版本和复核包。

## 提问规则

只在答案会改变需求含义、项目归属、核心技术落点、公开契约、数据结构、文件清单或实现效果时提问。优先一次列出相关阻断。

继续取证而不提问的情况包括：本地代码可以确认、只是章节或措辞偏好、现有代码已有唯一惯例、用户未指定文档文件名且只要求对话输出。

## 最终回复

技术设计后附上：

```text
执行结果：
- 需求基线：
- 需求来源与上游复核状态：<上游文档 ID / 修订号 / PASS 或草稿；客户原始需求方案等无交接头材料标注“无上游复核状态”>
- 代码基线：
- 项目一致性：
- 代码充分性：
- 文档位置：对话输出 / 文件路径
- 预计修改、新增、删除文件数量：
- 独立复核：PASS / 未执行
- 复核轮次与问题闭环：
- 待确认项：无 / 列表
- 已披露证据边界：
```

受限版本必须在标题附近标记复核限制。业务或材料阻塞时，只输出阻断报告，不包装为可实施方案。
