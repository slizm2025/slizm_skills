# 根据需求与代码生成技术设计

以需求定义目标，以当前实际生效的代码定义落地约束。只产出技术设计，不修改业务代码、配置或测试。

## 结果状态

- `READY`：设计完成需求覆盖、代码取证、影响分析、追溯和作者自检，完整且可实施。
- `LIMITED`：设计已完成作者自检，但存在关键非阻断证据限制，或用户明确只要受限草稿。
- `NEEDS_INPUT`：项目不匹配、关键代码缺失、需求冲突或核心设计存在会改变结果的多种合理解释。

只有 `READY` 可以使用“可直接实施”或等价表述。`LIMITED` 必须标明实际证据限制；`NEEDS_INPUT` 只输出阻断报告。是否另行审查不影响本 skill 的结果状态。

## 输入与阶段边界

独立识别并组合以下任一输入，不要求其他 skill 或固定交接头：

- 正式需求文档或 PRD；
- 普通需求说明；
- 原始附件、图片、原型或对话原文；
- 已有技术设计，用于核验、补全或更新。

文档 ID、版本、已有外部审查记录、需求基线和代码基线是可选审计元数据。存在时原样保留并对当前材料核验；缺失时自行建立本次需求和代码基线，不要求补做审查。按内容语义判断权威性，不按材料来自哪个 skill 判断，也不让来源文档的状态自动决定本设计的结果状态。

本 skill 不执行代码实现。用户在同一请求中同时要求设计和实现时，先完成本设计阶段；宿主可以在当前请求中进入独立实施阶段，无需要求用户重复授权。

## CodeGraph 代码路由

需要理解、定位或评估代码影响时，在首次代码搜索或源码读取前完整读取并执行 [CodeGraph 代码取证协议](references/codegraph-integration.md)。确认目标业务项目根目录并检查其 `.codegraph/`：存在时执行 CodeGraph-first；不存在或能力不可用时记录状态并用普通取证建立等价的一手证据。CodeGraph 不替代源码核验、项目身份验证或作者自检。

## 工作流

### 1. 建立本次基线

完整读取全部需求材料、补充关系、当前代码和已有技术设计。图片和图表使用视觉读取能力查看原件。记录来源定位、可用版本、用户明确关系、项目范围和代码快照。

已有稳定需求编号原样保留。仅给没有编号的规范性需求分配未占用的 `REQ-nnn`。已有技术设计中的稳定 `DES-nnn` 在决定含义未改变时保留；每个尚无编号的技术决定分配未占用的 `DES-nnn`。

本步骤完成条件：每个规范性需求和技术决定都有稳定编号，或已登记为阻断项；可选元数据已继承或本次基线已建立。

### 2. 核验项目与代码充分性

读取并执行 [项目一致性与代码充分性门禁](references/project-and-code-gates.md)。只使用目标业务项目根目录的 `.codegraph/`，记录路由状态、分支或 commit 及索引新鲜度。旧基线过期时重核需求触及范围，不因无关变化阻断。

结论不是“项目一致，代码完整或部分完整但足以设计”时转为 `NEEDS_INPUT`。

本步骤完成条件：项目一致，CodeGraph 路由已完成或明确不适用，且每条需求具有建立技术落点所需的项目证据。

### 3. 建立 CodeGraph 语义基线

为每条 `REQ` 建立查询包；小型相关需求可以合并查询，但结果必须映射回各自编号。至少回答：

- 当前从哪里进入，哪些符号负责主要行为；
- 数据、状态、配置、schema 和外部服务如何参与；
- 接口有哪些实现，调用和依赖方向是什么；
- 哪些抽象可以复用，哪些测试证明当前行为；
- 候选修改符号有哪些调用者、消费者和影响面；
- 哪些范围未被索引、结果不完整或已经过期。

先查询整体调用链，再按需窄化到接口实现、数据关系、测试、复用点和消费者。读取最终候选文件的关键源码；CodeGraph 未覆盖的配置、生成源、迁移或语言使用直接读取补充。

维护内部矩阵：

| 需求 | 当前语义路径 | 一手代码证据 | 设计变化 | 计划符号/文件 | 影响面 | 验证 |
|---|---|---|---|---|---|---|

本步骤完成条件：每个 `REQ` 都有足以选择技术落点的当前语义证据，或已有精确阻断说明。

### 4. 形成 `REQ → DES` 设计

每个 `DES-nnn` 必须映射需求，给出当前语义路径和一手代码证据，说明技术变化、输入处理输出与副作用、预计符号和文件、影响面、保持不变项及验证落点。

计划修改现有文件时，必须有 CodeGraph 语义证据，或属于未索引范围并有直接读取证据。计划新增文件时，说明现有符号不能承载的原因、目录惯例和新文件接入现有调用链的方式。

新增跨模块依赖、修改公共接口或持久化结构、增加计划外模块、复用点失效或调用方向改变时，重新执行相关 CodeGraph 查询或降级取证并刷新矩阵。

本步骤完成条件：不存在孤立 `DES`、无依据文件、未核验影响面或未映射验证，每条需求都有闭合设计路径。

### 5. 生成或更新技术设计

读取 [技术设计结构与输出协议](references/technical-design-template.md)，按复杂度选择轻量或标准结构。更新旧设计时保留仍有效的编号和元数据，明确新增、修改、废弃的 `DES`，并说明文件清单变化。

用户要求写入文件时先检查目标；来源或覆盖意图不明的同名文件不直接覆盖。默认文件名为 `<需求主题>-技术设计.md`。

本步骤完成条件：需求、`DES`、语义路径、一手代码证据、预计文件、影响面、实施依赖和验证互相闭合。

### 6. 作者自检

逐项检查需求覆盖、代码真实性、现状与目标分离、最小变化、编号稳定性、文件清单、实施依赖、验证映射和基线有效性。确认应调用时已 CodeGraph-first，结果已核验到当前源码，未索引、失败和过期范围已披露，没有遗漏调用者、实现类或更合适的现有复用点。

本步骤完成条件：每个 `REQ`、`DES`、预计文件、影响面和验证项均已核对，未解决的实质问题已转为 `NEEDS_INPUT`。

## 审查边界

本 skill 在作者自检和设计交付后结束。不得把启动子 agent、独立审查实例或审查修订循环作为完成步骤，也不得因为没有审查实例而把 `READY` 降级为 `LIMITED`。

是否审查由使用者在交付后自行决定。需要审查时，在新的 agent 或 task 中单独发起，并按审查目标提供最终设计、必要需求材料和代码范围；审查过程及结论不由本 skill 自动创建、等待或闭环。

## 最终回复

```text
执行结果：READY / LIMITED / NEEDS_INPUT
- 需求来源与可选元数据：
- 需求基线：
- 代码基线：
- CodeGraph：路由状态、目标索引根目录与新鲜度限制
- 保留及新增的 REQ/DES：
- 项目一致性与代码充分性：
- 文档位置：对话输出 / 文件路径
- 预计修改、新增、删除文件数量：
- 代码实施：未由本 skill 执行
- 后续审查：未自动执行；是否另开 agent 由使用者决定
- 待确认项：无 / 列表
- 已披露证据边界：无 / 列表
```

`LIMITED` 在标题附近标记实际证据限制。`NEEDS_INPUT` 不包装为可实施方案。
