## 1. 候选代码单元

先观察工作区实际组织方式。目标可能是单体、多模块、多仓库、公共库、命令行工具、桌面或移动应用、插件、数据流程、声明式配置、基础设施定义或其他形态。

根据实际存在的内容识别候选项：项目、组件、模块、包或命名空间；构建与依赖清单、入口、公开契约、配置和资源；需求中的专有术语、标识符、路径、命令、协议和数据结构；输入输出、状态变化、持久化、生成过程和外部依赖；与其他代码单元或环境的关系。

这些是候选维度，不是固定技术分层。工作区包含多个独立代码单元时分别列出。

## 2. 项目一致性

分别记录用户指定范围、本地识别出的代码身份和一致性结论。用户指定范围说明分析位置，不能覆盖明确的身份冲突。

### 一致

一个决定性证据能够唯一对应需求和代码，或多个相互独立的强佐证共同建立关系，且没有反向证据。有效证据包括唯一标识、公开契约、实际生效路径、项目元数据，以及多个专有对象、字段和规则的共同吻合。

目录名相似、单个通用词语相同、待审文档声称某路径存在或 `.codegraph/` 存在本身不构成充分证据。

### 不一致或无法确认

项目用途、业务域、公开契约、核心标识或行为存在明确冲突时为不一致；存在多个同样合理候选或证据不足时为无法确认。结论不是“一致”时停止设计并说明需求指向、本地识别、缺失证据和需要补充的内容。

## 3. CodeGraph 项目边界

先确认目标业务项目根目录，再检查该根目录的 `.codegraph/`。Skill 仓库、父工作区或其他仓库的索引不能证明目标项目。

CodeGraph 返回的入口、专有符号、调用关系和测试可以辅助身份判断，但必须结合构建文件、包结构、项目元数据、公开契约、远程信息或需求材料。索引路径、分支或 commit 与目标代码不一致时，不使用其结果支撑设计。

项目一致后按 [CodeGraph 代码取证协议](codegraph-integration.md) 选择 MCP、CLI 或普通取证。没有索引或能力不可用只改变证据方式；缺少的语义证据会改变技术落点时才阻断设计。

## 4. 从需求推导必需代码事实

项目一致后，逐条需求确定设计必须知道：

- 当前入口和主要行为符号；
- 接口实现、调用者、被调用者和依赖方向；
- 数据、状态、配置、资源、schema 与外部依赖来源；
- 结果输出、保存、传递与副作用；
- 证明当前行为的测试、契约、配置或部署定义；
- 可复用抽象、候选修改位置和主要影响面。

只检查会影响技术落点、文件清单、公开契约、数据规则或验证策略的内容。

## 5. 权威实现来源

优先读取实际生效的运行时代码和调用关系、声明式配置和基础设施定义、schema 和生成来源、环境实际消费的配置、数据转换和工作流，以及证明行为的测试。

CodeGraph 用于发现符号和关系。返回完整当前源码时可以作为读取证据；返回摘要或不完整关系时继续读取源文件。README、注释、历史设计和示例是辅助证据。与实际生效来源冲突时记录冲突，并判断是否阻断设计。

每个代码事实至少记录 CodeGraph 路由状态、语义路径、定位、符号或配置项、当前职责、实际关系、需求关系和证据边界。

## 6. 代码充分性

### 完整

能够确认需求涉及的现状流程、核心规则、事实来源、执行关系、技术落点、影响面、必要边界和现有验证惯例。

### 部分完整但足以设计

只有同时满足以下条件才能使用：

1. 文档中的每个当前事实都有实际证据。
2. 缺失内容不会改变业务效果、公开契约、数据规则、核心职责或主要文件清单。
3. 验证策略仍可准确落到层级或目录。
4. 文档没有把未分析范围伪装成完整影响范围。
5. 缺口及 CodeGraph 降级或新鲜度限制已披露。

### 关键代码缺失

缺失使模块职责、接口或数据契约、当前调用关系、核心文件、影响面或验证落点只能猜测时，停止设计并说明已确认事实、缺失事实、受影响需求和需要补充的位置。

## 7. 版本边界

记录仓库、分支、commit、tag、版本、文件状态、索引状态或分析时点。需求或相关代码变化后重新核验受影响证据。输入材料记录的代码基线与当前代码不一致时标记过期，只在相关代码变化或影响无法判定时阻断。索引与源码冲突时以直接读取为准并标记过期。
