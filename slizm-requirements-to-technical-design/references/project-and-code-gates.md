## 1. 候选代码单元

先观察工作区实际组织方式。目标可能是单体、多模块、多仓库、公共库、命令行工具、桌面或移动应用、插件、数据流程、声明式配置、基础设施定义或其他形态。

根据实际存在的内容识别候选项：

- 项目、产品、组件、仓库、模块、包或命名空间名称；
- 构建与依赖清单、入口、公开契约、配置和资源；
- 需求中的专有术语、标识符、路径、命令、协议和数据结构；
- 输入输出、状态变化、持久化、生成过程和外部依赖；
- 与其他代码单元、系统或环境的关系。

这些是候选维度，不是固定技术分层。工作区包含多个独立代码单元时分别列出。

## 2. 项目一致性

分别记录用户指定范围、本地识别出的代码身份和一致性结论。用户指定范围说明分析位置，不能覆盖明确的身份冲突。

### 一致

满足以下任一条件，且没有反向证据：

- 一个决定性证据能够唯一对应需求和代码；或
- 多个相互独立的强佐证共同建立对应关系。

有效证据包括：

- 需求中的唯一标识、公开契约或资源在实现中存在；
- 需求入口和相关行为形成实际生效路径；
- 项目元数据明确声明相同用途、产品或组件；
- 多个专有对象、字段和规则在同一代码单元中吻合。

目录名相似、单个通用词语相同或待审文档声称某路径存在，不构成充分证据。

### 不一致

项目用途、业务域、公开契约、核心标识或可观察行为存在明确冲突。

### 无法确认

存在多个同样合理的候选项，或证据不足以唯一定位。

结论不是“一致”时停止设计，使用以下结构提问：

```text
当前无法确认需求与本地代码属于同一目标：
- 用户指定范围：
- 需求指向：
- 本地识别：
- 不匹配或缺失证据：
- 需要用户提供：
```

## 3. 从需求推导必需代码事实

项目一致后，逐条需求确定设计必须知道的事实：

- 当前由什么入口或使用方式触发？
- 哪些实现、声明或生成来源负责关键判断、转换、编排和状态变化？
- 数据、状态、配置、资源和依赖来自哪里？
- 结果如何输出、保存、传递或产生副作用？
- 哪些契约、测试、配置或部署定义证明当前行为？
- 需求变化应落在哪个职责边界，哪些行为需要保持？

只检查会影响技术落点、文件清单、公开契约、数据规则或验证策略的内容。

## 4. 权威实现来源

优先读取实际生效的来源，而不是按文件类型预设权威性：

- 运行时代码和真实调用关系；
- 声明式配置和基础设施定义；
- schema、契约、规则文件和生成来源；
- 构建、部署或运行环境实际消费的配置；
- 数据转换、模型和工作流定义；
- 能证明行为的测试和固定样例。

README、注释、历史设计和示例是辅助证据。与实际生效来源冲突时记录冲突，并判断冲突是否阻断设计。

每个代码事实至少记录：定位、符号或配置项、当前职责、实际关系、与需求的关系和证据边界。

## 5. 代码充分性

### 完整

能够确认需求涉及的现状流程、核心规则、事实来源、执行关系、技术落点、必要边界和现有验证惯例。

### 部分完整但足以设计

只有同时满足以下条件才能使用：

1. 文档中的每个当前事实都有实际证据。
2. 缺失内容不会改变需求业务效果。
3. 缺失内容不会改变公开契约、数据规则、核心模块职责或主要文件清单。
4. 验证策略仍可准确落到层级或目录。
5. 文档没有把未分析范围伪装成完整影响范围。
6. 缺口已作为证据边界披露。

### 关键代码缺失

缺失会使以下任一内容只能猜测：

- 模块或职责归属；
- 接口、消息、数据或持久化契约；
- 当前规则、调用关系或数据来源；
- 核心文件清单或验证落点；
- 需求实施效果。

关键代码缺失时停止设计，说明已确认事实、缺失事实、受影响需求和需要补充的位置。

## 6. 版本边界

记录可获得的仓库或代码单元、分支、commit、tag、版本、文件状态或分析时点。

需求、相关代码或待审技术文档变化后，重新核验受影响证据。任何文档修改都会使旧独立复核结论失效。
