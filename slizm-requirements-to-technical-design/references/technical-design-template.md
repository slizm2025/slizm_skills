## 1. 结构与元数据

默认使用轻量结构。涉及多个服务或端、公开接口、消息契约、持久化模型、迁移、复杂状态流转或较大文件范围时，展开标准结构。省略空章节，不让模板扩大需求。

文档 ID、版本、已有外部审查记录、需求基线和代码基线是可选审计元数据，不是固定交接头，也不是本 skill 的完成条件。已有值原样保留并核验；缺失时记录本次可获得的基线，不补造历史信息，也不自动启动审查。

已有稳定 `REQ` 和 `DES` 原样保留。每个新增技术决定分配未占用的 `DES-nnn`，并映射至少一条需求。

- 需求来源与可选审计元数据：

- 代码基线：

## 2. 现状依据与整体实现

## 3. 技术决定

每个技术决定都使用以下语义结构，不得省略映射和验证：

```text
DES-001
├─ 对应需求：REQ-001
├─ 当前代码证据
├─ 技术变化
├─ 输入、处理、输出、副作用
├─ 预计文件
├─ 保持不变项
└─ 验证落点
```

## 3. 标准结构扩展

在轻量结构内按需增加：

- 跨端或跨服务调用链；
- 状态机和关键数据流；
- API、RPC、事件或消息契约；
- 数据模型、持久化和迁移；
- 配置、权限或外部集成；
- 分层验证策略和回滚条件；

只有需求和代码证据涉及相应变化时才增加。

## 4. 文件与实施依赖规则

文件清单可以包含预计修改、新增、删除和为解释关系而列出的复用文件。现有文件必须真实存在；拟新增路径必须符合当前目录和同类命名惯例；关联代码不等于修改范围。

每个预计变更文件必须映射至少一个 `DES` 和一个需求编号。每个 `DES` 的全部预计文件必须出现在文件清单中。无法确定准确文件名时，给出目录、命名依据和待确认状态。

实施顺序按真实依赖给出。`依赖的 DES` 只记录前置技术决定；无依赖时明确为“无”。不得用文档章节顺序替代实现依赖，也不得出现矩阵中没有定义的 `DES`。

## 5. 验证与生命周期

设计阶段说明验证什么、在哪一层验证：已存在的测试文件可直接引用；有充分惯例的拟新增测试文件标记为预计新增；证据不足时给出测试层级、目录、场景和实施阶段确认项。验证只覆盖需求、契约、`DES` 和明确保持不变项。

完整追溯必须满足：

```text
REQ → DES → 代码证据 → 文件 → 验证
```

需求、相关代码或已有设计决定变化时，重新核验受影响的追溯、文件清单和验证。维护旧设计时说明保留、新增、修改和废弃的 `DES`，以及受影响文件与验证。

最终执行结果格式以主 `SKILL.md` 为单一来源。
