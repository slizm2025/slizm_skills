## 1. 适用结果

本协议评价完整技术设计，不继续共同设计，也不直接修改文档。复核者从原始需求和本地代码重新取证，检查待审文档是否忠实、真实、闭合且可实施。

## 2. 前置条件与隔离

复核前必须具备：

- 全部有效需求文件及用户明确补充、修正和撤回原文；
- 可访问的本地项目范围和代码快照信息；
- 完整待审技术设计；
- 一个全新、未参与主分析且能只读搜索项目的隔离审查实例。

运行时适配说明：在 Claude Code 等支持子代理的环境中，可用限定为只读工具集的全新子代理充当审查实例；其他环境以实际上下文隔离能力为准。

可以向复核者提供：

- 原始需求定位符或逐字正文及其明确关系；
- 工作区、用户指定范围和代码快照；
- 完整待审文档；
- 本协议的固定任务和报告格式。

保持以下作者材料在隔离边界之外：

- 作者提取的需求基线、追溯矩阵和代码结论；
- 作者的搜索记录、自检结果和预期 verdict；
- 之前审查者的问题清单或结论；
- 与需求无关的历史对话和偏好。

若一手材料不能完整传递，不能把审查称为完整独立复核。

## 3. 固定复核任务

使用以下中性任务，替换定位符但不加入倾向性结论：

```text
你是一名首次接触作者分析过程的独立技术设计审查者。本次审查只读，不修改、创建、删除或格式化任何文件，也不运行会改变项目状态的命令。

事实材料：
1. 有效原始需求：<定位符或逐字正文及明确关系>
2. 本地项目：<工作区和用户指定范围>
3. 代码快照：<branch/commit/version/时点>
4. 待审技术设计：<完整正文或稳定定位符>

请从头独立完成：
A. 提取全部规范性需求和保持不变项，并建立自己的需求编号。
B. 独立识别需求指向的项目，判断项目一致性和代码充分性。
C. 重新搜索实际生效的实现来源，不信任待审文档中的路径、符号和调用关系。
D. 逐项检查需求覆盖、当前代码事实、技术推导、整体调用和数据链路、文件清单、实施顺序及验证落点。
E. 检查是否遗漏、改变或扩大需求，是否虚构现状或文件，是否加入无依据重构、优化、依赖、兼容或产品规则。

每个问题必须包含：级别、文档位置、需求依据、代码证据、问题影响和不扩大需求的最小修正方式。没有足够证据时标记 UNVERIFIED，不把推测写成问题事实。只输出审查报告，不修改文档。
```

## 5. Verdict 规则

### PASS

只有全部满足才输出：

1. 所有有效需求均可访问并已审阅。
2. 项目一致性成立，代码完整或部分完整但足以设计。
3. 所有需求、保持不变项和验收结果都有准确设计落点。
4. 所有当前事实、现有文件、符号和调用关系都有权威实现证据。
5. 文件清单、整体方案、实施顺序和验证映射互相一致。
6. `BLOCKER=0`、`MAJOR=0`、`UNVERIFIED=0`，且没有影响准确性的未解决 `MINOR`。
7. 报告绑定的需求、代码和文档版本仍有效。

### FAIL

现有材料足够，但文档存在可以根据证据修正的问题。

### NEEDS_INPUT

必须由用户或外部来源补充、恢复或裁决材料，例如需求冲突、项目无法确认、关键代码缺失或核心设计存在多个会改变结果的合理解释。

## 6. 问题闭环与失效

作者逐条核验审查证据：

| 问题编号 | 处理 | 修订位置 | 需求或代码依据 |
|---|---|---|---|

- 证据成立：执行满足需求所需的最小修订。
- 证据不成立：保留文档并记录具体反证。
- `NEEDS_INPUT`：停止交付并提出聚焦问题。
- `FAIL`：文档保持非最终状态。

任何被复核载荷修订后，使用另一个全新隔离审查实例重新复核完整文档。需求来源、代码快照或被复核载荷内容变化都会使旧 `PASS` 失效；复核完成后据实写入复核证明元数据（复核状态、复核结论、复核绑定）不属于载荷修订，不使 `PASS` 失效。

完整复核累计不超过 3 轮。超过上限仍未 `PASS` 时，不再自动进入新一轮，转为 `NEEDS_INPUT`，向用户报告未收敛的问题清单并请求裁决。

## 7. 能力不可用时

没有满足隔离条件的审查能力时，可以输出受限版本，但必须：

- 在标题附近标记“独立复核未执行”；
- 说明原因和准确性限制；
- 将自检称为作者自检；
- 附上可移交复核包：需求基线、项目路径、代码快照、完整文档和本协议固定任务；
- 不使用“通过”“问题闭环完成”或等价表述。

缺少审查能力只降低复核状态。需求或关键代码缺失、项目不一致和未解决业务阻塞仍会阻止输出可实施方案。
